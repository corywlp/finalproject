⚑ Why HIPAA Matters Here

HIPAA (the Health Insurance Portability and Accountability Act) is a federal law that requires any organization handling patient health data to keep it private and secure. ClearPath is legally required to comply.


If PHI is stolen or exposed, HIPAA requires ClearPath to:

  • Report the breach to the U.S. Department of Health & Human Services (HHS)
  • Notify every affected patient in writing within 60 days
  • Publicly announce the breach if more than 500 patients are affected

Penalties can reach $1.9 million per violation category per year. This is why data exfiltration — data leaving the network without authorization — is treated as a Critical severity incident in healthcare environments.

Resources & Help

Use this page if you get stuck. It won't tell you what to find — but it will help you understand what you're looking at and fix common technical problems.

Common Errors & Fixes

SIEM — "Database not loaded yet"

The SIEM requires an internet connection to load the database engine. Check your connection and refresh the page. If you're in a restricted network environment, ask your instructor.

SIEM — Query returns no rows when you expect results

Check your spelling, capitalization, and quotes. SQLite string comparisons are case-sensitive. Try SHOW TABLES; first to confirm the database loaded correctly.

Python — python: command not found

Try python3 instead of python. On Windows, try py. If none work, Python may not be installed — ask your instructor.

Python — FileNotFoundError: auth.log

Your script and auth.log must be in the same folder. In your terminal, use cd to navigate to that folder before running the script. Use ls (Mac/Linux) or dir (Windows) to confirm both files are there.

Python — Script runs but prints nothing

This usually means your filtering condition isn't matching anything, or your threshold is set too high. Add a temporary print(line) inside your loop to verify lines are being read. Check what you're searching for against actual lines in the file.

PDF export — content gets cut off

In your browser's print dialog, set margins to "Minimum" or "None" and make sure "Background graphics" is enabled. If text fields are too long, they may wrap across pages — that's normal.

Glossary

Terms you may encounter during this investigation.

Brute Force Attack

Trying many passwords rapidly until one works. Detected by large volumes of failed login attempts from the same source in a short window.

Persistence

How attackers maintain access after the initial intrusion — even through reboots or password changes. Common methods: scheduled tasks, backdoor accounts, startup scripts.

Cron Job

A scheduled task on Linux that runs automatically at set intervals. Stored in /etc/cron.d/. Attackers use these to re-run malicious code without being present.

Reverse Shell

The compromised machine calls out to the attacker's server, giving them remote control. Bypasses inbound firewall rules because the connection goes outbound.

C2 / Command & Control

A server the attacker controls that receives connections from compromised machines. The infected system "beacons" out to receive instructions or send stolen data.

Exfiltration

Unauthorized transfer of data out of a network. Look at what was queried from the database and where outbound connections went.

IOC — Indicator of Compromise

Evidence of an attack: suspicious IPs, unexpected accounts, unusual processes, modified files, or anomalous network traffic.

Privilege Escalation

Gaining higher permissions than originally obtained. On Linux, this means getting root access. Watch for accounts added to the sudo group.

Anti-Forensics

Techniques to hide activity or hinder investigation — deleting logs, clearing shell history, renaming malicious files to look legitimate.

Backdoor Account

An account created by an attacker to maintain access. Often named to blend in with legitimate service accounts. Look for accounts created at odd hours with elevated privileges.

SSH & Authorized Keys

SSH is the protocol for remote server access. Keys stored in ~/.ssh/authorized_keys allow passwordless login — if an attacker adds their key there, they can return anytime.

Package Manager / dpkg

Linux software installs are logged in /var/log/dpkg.log. Unexpected software installed at odd hours is a red flag.

Process Log / Audit Trail

Records what commands ran on a system, by whom, and when. Different from auth logs (which track logins) — process logs show what happened after someone was already in.

Orphaned Account

An account left active after someone no longer needs access — former employees, expired contractors. A common and overlooked attack vector.

MFA — Multi-Factor Authentication

Requiring a second verification step beyond a password — like a one-time code. A stolen password alone cannot complete login if MFA is enabled.

SIEM

Security Information and Event Management — a system that aggregates logs and lets analysts search for security events. A searchable record of everything that happened on the network.

Root / sudo

root is Linux's all-powerful admin account. sudo lets a regular user run commands with root privileges. Any account in the sudo group has full system control.

Firewall

Filters network traffic by IP, port, or protocol. Logs both allowed and blocked connections — outbound traffic to unusual IPs can signal a breach.

Least Privilege

Users and systems should only have the minimum access needed. Over-provisioned accounts give attackers more power when compromised.

PHI — Protected Health Information

Health data that can identify a person — names, DOBs, diagnoses, insurance IDs, billing records. Protected under HIPAA.

NIST Cybersecurity Framework

Five functions for managing security risk: Identify, Protect, Detect, Respond, Recover. Used to structure incident response recommendations.

Incident Response Plan

A documented procedure for detecting, containing, and recovering from a breach. Only useful if it's current and people know it exists.

SQL

The language used to query databases. SELECT retrieves data. In a breach, DB query logs show exactly what was accessed and by whom.

Phishing

Impersonating a trusted entity to trick someone into giving up credentials or clicking a malicious link. Not how this breach started — but a common initial access method.

Ransomware

Malware that encrypts files and demands payment. Often deployed after persistence and data theft are already complete.

Network Segmentation

Dividing a network into isolated zones so a breach in one area doesn't expose everything. A flat network gives an attacker access to all systems from one entry point.

Backup & Disaster Recovery

Regularly copying data to a separate location so it can be restored after an incident. A backup is only useful if it has been tested — an untested backup may fail when you need it most.

HIPAA Breach Notification — What You Need to Know

You will need to address HIPAA obligations in your final report. Here are the key facts — you'll need to apply them to this specific incident yourself.

What triggers notification? A breach of unsecured PHI that has not been shown to have a low probability of compromise. If an unauthorized party accessed systems containing PHI, notification is presumed required unless proven otherwise.
Who must be notified? Affected individuals must be notified. If 500+ individuals in a state are affected, prominent media outlets in that state must also be notified. The Department of Health & Human Services (HHS) must always be notified.
When? Notification to individuals and HHS must occur within 60 calendar days of discovering the breach. There is no minimum — earlier is better.
Your job in the report: Based on what you found, argue whether this incident meets the threshold for mandatory notification — and why. Don't just say "yes" or "no" — support it with your findings.

NIST CSF Mapping

For each of the five functions, ask: what did ClearPath do (or fail to do) in this area? Use your interview with Jordan to inform Identify and Protect. Use your SIEM and terminal findings for Detect. Your playbook covers Respond and Recover.

Recommendations

Be specific. "Improve security" is not a recommendation. "Implement MFA on all remote access methods within 30 days" is. Immediate recommendations should address the active threat. Short-term closes the gaps. Long-term builds a sustainable posture.

Ethics Section

Think about your role as the analyst. You have access to sensitive data, logs, and systems. What obligations come with that? Consider confidentiality, scope of your investigation, and what you do if you discover something unexpected.

NIST CSF Quick Reference

IDENTIFY

Understand your assets, risks, and environment. Who has access to what? What data do you hold? What are your most critical systems?

PROTECT

Implement safeguards to limit the impact of an incident. Access controls, MFA, encryption, training, and patching all live here.

DETECT

Identify when an incident is occurring. Logging, monitoring, alerting, and anomaly detection. Ask: how quickly was this attack noticed?

RESPOND

Take action during and after an incident. Containment, eradication, communication, and execution of the IR plan.

RECOVER

Restore systems and operations. Validate that systems are clean before bringing them back online. Document lessons learned.

INCIDENT REPORT — INC-2026-0041

Analyst: Alex Chen, SOC II Date: March 3, 2026 Client: Meridian Retail Group Classification: Confidential — Internal Use Only Incident Type: Business Email Compromise / Financial Fraud Severity: High

Executive Summary

On February 28, 2026, a targeted phishing campaign against Meridian Retail Group's accounts payable department resulted in the theft of an employee's Active Directory credentials and subsequent unauthorized access to the company's internal invoicing system. The attacker, operating from IP address 91.108.4.200, used the stolen credentials to authenticate to the corporate VPN and gain access to the invoicing platform, where they modified two vendor payment records to redirect a combined $42,000 to an external bank account under their control.

The breach went undetected for approximately 22 hours. Although a SIEM alert fired on the anomalous VPN login within minutes of the attacker's entry, it was not reviewed until the following morning — at which point the financial team had already flagged the discrepancy through routine reconciliation. The compromised account was disabled and the VPN session terminated by 9:45 AM on March 1, approximately 6 hours after detection.

A wire recall was initiated through Meridian's banking partner on March 1. As of the date of this report, $28,000 has been recovered. The remaining $14,000 is being pursued through the bank's fraud resolution process. No customer personal data, payment card data, or other regulated information was accessed during the incident. This report documents the full timeline, technical findings, control failures, and recommended remediation actions.

Attack Timeline

Feb 28, 10:14 AM — A phishing email purportedly from Meridian's IT helpdesk arrived in the inbox of J. Torres, an accounts payable specialist. The email claimed that Torres's account password was expiring and directed them to a spoofed login page hosted at meridian-it-support[.]com — a domain registered just four days earlier. Torres entered their Active Directory username and password on the spoofed page.

Feb 28, 10:31 AM — Attacker authenticates to Meridian's corporate VPN from IP 91.108.4.200 using Torres's stolen credentials. A SIEM alert fires but is queued for next-morning review with no real-time escalation configured.

Feb 28, 10:44 AM — Attacker connects to the invoicing platform using Torres's account and gains full access to all vendor payment records.

Feb 28, 11:02 AM — Attacker modifies two pending vendor payment records, redirecting $42,000 to an attacker-controlled account.

Feb 28, 11:09 AM — Attacker logs out of VPN. Total dwell time: approximately 38 minutes.

Mar 1, 08:15 AM — Finance team flags discrepancies during morning reconciliation. IT Security notified immediately.

Mar 1, 08:22 AM — Security team finds the overnight SIEM alert. Torres's account disabled; incident formally opened.

Mar 1, 09:45 AM — Incident declared contained. Wire recall initiated. Payment records restored from backup.

Findings

Finding 1 — Credential Phishing via Spoofed Domain

The attacker registered meridian-it-support[.]com four days before the attack and hosted a replica of Meridian's IT portal. The email used urgency language to pressure the recipient into acting quickly. J. Torres, who had not received phishing awareness training since onboarding in 2024, did not recognize the spoofed domain. This is the initial access vector for the entire incident.

Finding 2 — No Multi-Factor Authentication on VPN

Meridian's VPN required only a username and password. Once credentials were stolen, no additional verification prevented access. Had MFA been enabled, the attacker would have been blocked even with valid credentials — this single control failure converted a phishing success into full network access.

Finding 3 — Excessive Account Privileges

Torres's role required access to a specific subset of vendor payments, but their account had read/write access to all vendor payment records system-wide. Had least-privilege principles been enforced, the blast radius of this incident would have been significantly smaller.

Finding 4 — SIEM Alert Not Actioned in Real Time

The SIEM correctly flagged the anomalous VPN login within 17 minutes. However, no escalation policy existed for this alert type during non-business hours. The SIEM did its job; the response process did not.

NIST CSF Analysis

Identify: Meridian had not classified its invoicing system as a high-risk asset despite processing over $2M/month in vendor payments. Access controls did not reflect the sensitivity of the data.

Protect: MFA was absent on VPN, least-privilege was not enforced in the invoicing system, and phishing training had not been refreshed since onboarding. Each failure independently contributed to the outcome.

Detect: The SIEM generated a valid alert within minutes. Detection tools were in place; detection processes were not — no escalation path existed for off-hours medium-severity alerts.

Respond: Once notified, the response was fast and effective — account disabled within 7 minutes, banking partner contacted within 2 hours. The problem was the 22-hour delay in triggering the response.

Recover: Payment records restored from nightly backup with no data loss. $28,000 of $42,000 recovered as of this report. No systems required rebuilding.

Recommendations

Immediate (0–7 days): Enable MFA on all VPN accounts. Force a password reset for potentially targeted accounts. Audit invoicing system permissions and revoke any access exceeding role requirements. Preserve all logs in tamper-evident storage.

Short-term (1–30 days): Implement real-time escalation policy for off-hours SIEM alerts. Configure alerts on payment record modifications. Launch company-wide phishing awareness training. Implement DMARC, DKIM, and SPF on Meridian's email domains.

Long-term (30–90 days): Extend MFA to all systems handling sensitive data. Establish formal asset classification policy. Implement periodic access recertification every 6 months. Consider engaging an external red team after above changes are implemented.

Legal & Compliance Considerations

No customer personal data, payment card numbers, or other regulated consumer information was accessed. The attacker's activity was confined to vendor payment records — commercial data — which does not trigger notification requirements under applicable state consumer breach notification laws or PCI-DSS. All incident documentation has been preserved. If the wire recovery process fails, Meridian's legal team is prepared to file a referral with the FBI's IC3.
